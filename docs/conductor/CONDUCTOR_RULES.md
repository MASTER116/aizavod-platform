## Раздел 29: CONDUCTOR — ПРАВИЛА ДЕКОМПОЗИЦИИ

> Дополнение к RULES MASTER v1.0, раздел 27

### 29.1 Трёхуровневая иерархия

Любая задача от основателя проходит через 3 уровня декомпозиции:

1. **CONDUCTOR → Директора** — стратегическое распределение по зонам ответственности
2. **Директора → Начальники отделов** — тактическое распределение по компетенциям
3. **Начальники отделов → Специалисты** — конкретные исполнительные задачи

### 29.2 Правила декомпозиции

1. **Минимум директоров.** Не привлекать директора, если задача его не касается. "Исправить баг в API" → только CTO, не нужен CFO.

2. **Автоматическая остановка.** Декомпозиция останавливается когда задача:
   - Может быть выполнена за < 2 часов
   - Не требует координации с другими отделами
   - Достаточно конкретна для немедленного исполнения

3. **Зависимости.** CONDUCTOR определяет порядок выполнения:
   - `SEQUENTIAL` — задача B ждёт завершения задачи A
   - `PARALLEL` — задачи выполняются одновременно
   - `BLOCKED` — задача ждёт внешнего ввода (от основателя, партнёра)

4. **Приоритизация по классам:**
   - Класс A (деньги/грант/запуск) → CRITICAL/HIGH
   - Класс B (репутация/open-source) → NORMAL/LOW

5. **Каждый директор обязан:**
   - Оценить свою часть задачи (feasibility)
   - Назвать конкретные deliverables
   - Указать зависимости от других директоров
   - Дать оценку времени

### 29.3 CEO как арбитр

Если между директорами конфликт приоритетов (CTO хочет рефакторинг, CMO хочет фичу для маркетинга):
- CEO принимает решение на основе принципа 4.3 (что ближе к деньгам — первое)
- Решение фиксируется в ConductorLog

### 29.4 Отчётность снизу вверх

```
Специалист: "Готово: написал 3 компонента React" → Начальник отдела
Начальник: "Frontend готов на 80%, осталось 2 компонента" → Директор
Директор: "CTO: технически лендинг готов, ждём дизайн от CDO" → CONDUCTOR
CONDUCTOR: "Лендинг готов на 60%: код готов, дизайн в работе, тексты готовы" → Основатель
```

### 29.5 Формат задачи на каждом уровне

```json
{
  "id": 42,
  "title": "Краткое название",
  "description": "Полное описание",
  "level": "director|head|specialist",
  "assigned_to": "cto|cto.frontend|cto.frontend.web_developer",
  "parent_id": 41,
  "priority": "critical|high|normal|low",
  "dependencies": [40, 39],
  "execution_order": "parallel|sequential|blocked",
  "deliverables": ["файл1.py", "деплой на сервер"],
  "estimated_hours": 2,
  "status": "pending|in_progress|completed|failed"
}
```

### 29.6 Быстрый путь (Fast Track)

Для простых задач CONDUCTOR может пропустить уровни:
- "Исправь баг в conductor.py" → сразу cto.backend.fastapi_dev (пропуск 2 уровней)
- "Сколько стоит сертификация?" → сразу certifier (пропуск всех уровней, роутинг)
- "Стратегия на месяц" → CEO декомпозирует → все директора

## Раздел 30: ДИРЕКТОРСКИЕ ПРОМПТЫ

Каждый директор имеет системный промпт, определяющий:
- Роль и зону ответственности
- Какие отделы ему подчиняются
- Как декомпозировать задачи
- Какие сервисы/файлы в его зоне
- Ограничения и правила

Промпты хранятся в `prompts/directors/` (YAML).
Отделы — в `prompts/departments/` (YAML).
Специалисты — генерируются динамически из контекста задачи.

## Раздел 31: МЕГА-ОРКЕСТРАТОР

CONDUCTOR может работать в двух режимах:

### 31.1 Режим роутера (текущий)
Простые вопросы → keyword match → один агент → ответ.
Время: мгновенно.

### 31.2 Режим оркестратора (новый)
Сложные задачи → CEO-декомпозиция → директора → отделы → специалисты → сборка.
Время: зависит от сложности.

Переключение автоматическое:
- Запрос-вопрос ("сколько стоит?") → роутер
- Запрос-задача ("сделай лендинг") → оркестратор
- Запрос-стратегия ("план на месяц") → оркестратор через CEO
